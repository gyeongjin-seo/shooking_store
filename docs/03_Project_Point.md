# 3. 어떻게 구조를 짤까? (설계도)

# 🏗️ 프로젝트 아키텍처 및 폴더 구조 설명서

본 문서는 `shooking_store` 프로젝트의 디렉토리 구조와 설계 의도를 설명합니다.
React의 컴포넌트 재사용성과 유지보수 편의성을 최우선으로 고려하여 **기능(Feature) 및 역할(Role) 중심**으로 구조를 설계했습니다.

---

## 1. 폴더 구조 개요 (Directory Structure)

```bash
src/
├── context/       # 전역 상태 관리 (Global State)
├── components/    # 재사용 가능한 UI 컴포넌트
│   ├── layout/    # 페이지 전체 틀 (Wrapper)
│   ├── common/    # 범용 컴포넌트 (버튼, 헤더 등)
│   └── product/   # 도메인 특화 컴포넌트 (상품 관련)
├── pages/         # 페이지 단위 조립 (View)
└── App.jsx        # 앱 진입점 및 Provider 설정
```

# 🏗️ Project Architecture & Design Decisions

본 문서는 `hooking_store` 프로젝트의 **폴더 구조 설계 의도**와 **핵심 UI/UX 구현 전략**을 기술합니다.
유지보수의 용이성과 확장성, 그리고 사용자 경험(UX)을 최우선으로 고려하여 설계되었습니다.

---

## 📂 1. 아키텍처 및 폴더 구조 (Structure)

### 1️⃣ Context API 분리 (`src/context/`)

> **설계 의도:** 장바구니 수량(`cartCount`)과 같이 앱 전역에서 접근해야 하는 데이터와 로직을 한곳에서 중앙 관리하기 위함입니다.

- **Prop Drilling 방지:** `App` → `Page` → `List` → `Item`으로 props를 불필요하게 깊이 전달하는 비효율을 제거했습니다.
- **로직 중앙화:** 장바구니 담기 로직(`addToCart`) 수정 시, 여러 컴포넌트를 탐색할 필요 없이 Context 파일 하나만 수정하면 됩니다.

### 2️⃣ 컴포넌트의 역할별 분리 (`src/components/`)

컴포넌트를 단순히 한 폴더에 모으지 않고, **성격과 재사용 범위**에 따라 세분화했습니다.

| 폴더명         | 역할 및 분리 이유                                                                            | 기대 효과 (Benefits)                                                                                |
| :------------- | :------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- |
| **`layout/`**  | **모바일 뷰 틀(Frame)**<br>콘텐츠와 무관하게 화면의 너비(`max-width`)와 정렬을 담당합니다.   | 모든 페이지에 동일한 모바일 뷰 환경을 강제할 수 있으며, 레이아웃 변경 시 이 파일만 수정하면 됩니다. |
| **`common/`**  | **공통 요소 (Universal)**<br>`Header`, `Button` 등 프로젝트 전역에서 재사용되는 UI입니다.    | 컴포넌트 재사용성을 높이고, 서비스 전체의 UI 일관성(Consistency)을 유지하기 쉽습니다.               |
| **`product/`** | **도메인 특화 (Feature-Specific)**<br>'상품'이라는 특정 기능에 강하게 결합된 컴포넌트입니다. | 상품 관련 기능 수정(예: 가격 포맷팅 변경) 시, 탐색 범위를 이 폴더로 좁혀 유지보수 효율을 높입니다.  |

### 3️⃣ 페이지와 컴포넌트의 분리 (`src/pages/`)

> **설계 의도:** `components`는 "화면을 그리는 부품(LEGO Blocks)"이고, `pages`는 "부품을 조립한 완성품(Model)"으로 정의합니다.

- **역할 명시:** `ProductPage.jsx` 파일만 보더라도 해당 페이지가 어떤 컴포넌트 조합(Header + ProductList)으로 이루어졌는지 한눈에 파악 가능합니다.
- **확장성:** 추후 `react-router-dom` 도입 시, 라우팅의 대상이 되는 페이지 단위가 명확해져 구조 확장이 용이합니다.

## 🎨 2. 핵심 UI/UX 구현 전략 (Implementation)

사용자 경험을 극대화하기 위해 적용한 기술적 전략입니다.

### 📱 모바일 퍼스트 레이아웃 (Mobile First)

- **구현:** Layout 컴포넌트에 `max-width: 480px`, `margin: 0 auto` 적용
- **이유:** PC 브라우저에서도 모바일 앱과 동일한 경험을 시뮬레이션하고, 콘텐츠가 가로로 과도하게 늘어지는 것을 방지하여 가독성을 높입니다.

### 🍱 CSS Grid 시스템 (ProductList)

- **구현:** `display: grid`, `grid-template-columns: 1fr 1fr`
- **이유:**
  - Flexbox보다 2차원 배치(행/열) 제어가 훨씬 용이합니다.
  - 미디어 쿼리 등 반응형 대응 시 코드 복잡도를 획기적으로 낮출 수 있습니다.

### 🖼️ 이미지 최적화 (ProductItem)

- **구현:** `aspect-ratio` 및 `object-fit: cover` 속성 사용
- **이유:** 원본 이미지의 사이즈나 비율이 제각각이어도, 화면에서는 일정한 비율(정사각형 등)로 깔끔하게 렌더링하여 시각적 완성도를 보장합니다.
